`src/views/home/ui/TopologyAnalysisBar.tsx` is the compact mode switcher shown over `/topology`.

It owns the visible Overview / Focus / Path / Health controls, the small metrics line, the Overview graph handoff brief, the Health stale/orphan/promotion chips, and the first actionable Health inspect target. The bar is reachable on mobile as a stacked control surface, hidden only in the intermediate tablet range where it would collide with the left workspace panel, and returns to the centered desktop overlay at the large breakpoint. When a selected-node drawer is open, the desktop bar reserves the drawer rail so mode actions remain clickable instead of sitting underneath the drawer. Path primer, progress, and result overlays move below that mobile bar while keeping their original desktop position.

Touch target contract: the compact bar keeps its primary mode tabs, Actions summaries, overview/health copy icons, and direct handoff links at a minimum 32px hit target. The topology map still reads as a dense graph workbench, but the actions that move evidence into Claude Code, Codex, `/ontology`, or the builder remain tappable on mobile and precise on desktop.

When Overview mode is active, the action panel is expanded by default and shows the proof order before any export: read the source-backed map, focus a graph handle, prove path evidence, then close the health gate. It can also copy a first-contact graph brief with total node and relation counts, stale/orphan/promotion health buckets, the current topology URL, a Health-mode URL, `/ontology/insights` handoff, `ontology-atlas overview [vault] --limit 5`, the MCP `query_ontology({ operation: "overview", limit: 5 })` payload, the `query_plan(targetOperation: "overview")` payload, and a workspace-brief check. The visible prompt now tells the user to read the source-backed ontology map before copying an agent handoff, so the default topology screen reads as graph evidence rather than a generic diagram. The copy action is a labeled compact CTA, not an icon-only affordance, so Claude Code / Codex handoff reads as an intentional next step. This keeps the first topology viewport from presenting an empty Actions rail and makes the default topology screen useful as an agent/collaborator handoff before a node is selected or a path is proven.

The Overview first viewport names those counts as ontology `concepts` and `relations`, not legacy project totals. The collapsed workspace card, the analysis metric row, and the canvas `application` label all use the same ontology-map language so a dogfood vault with one project shell still reads as a 240+ concept graph rather than a one-project project board.

Overview mode exposes agent-ready actions directly: copy a Claude Code / Codex ontology reanalysis command, copy the post-change update check, and keep the longer graph brief in the details section. Focus, Path, and Health modes still provide narrower MCP/CLI packets for selected nodes and relationship proof.

When Focus mode has a selected node, the bar first shows the review order: read the node profile, trace incoming impact, edit or confirm the meaning, then run the sync gate if the review turns into a change. It then exposes a focused review brief plus dedicated MCP profile, impact, and post-change sync-gate copy actions for that node. It also shows direct `/ontology` and `/ontology/edit` links beside the copy actions, so a focused visual neighborhood can move into browse or builder repair without requiring the user to paste the copied brief first. The review brief includes the selected node, the current topology URL, a focus URL that reopens the same node in Focus mode, `/ontology` and `/ontology/edit` handoff URLs, a read-only `ontology-atlas node <slug> [vault] --limit 12` CLI check, the matching MCP `node_profile` payload, incoming `blast_radius` impact checks, and the shared 14-check runtime graph DB plus health / cycles / growth / maintenance / validate sync sequence. The profile payload is `query_ontology({ operation: "node_profile", slug, depth: 2, limit: 12 })`; the impact payload is `query_ontology({ operation: "blast_radius", slug, depth: 2, direction: "incoming" })`. This lets a focused visual neighborhood become either a collaborator review artifact or an agent-ready node inspection, and it gives the user the same closing gate if that focus review turns into a builder edit.

When Health mode is active, the bar can copy a markdown health evidence brief. The brief includes total issue count, stale/orphan/promotion bucket counts, the first inspect target, the current URL, an inspect URL that reopens `/topology?mode=health` with the target node selected, an `/ontology` browse URL, a builder repair URL for the same target, a kind-specific next action, a read-only `ontology-atlas node <slug> [vault] --limit 12` CLI check, the matching MCP `query_ontology({ operation: "node_profile", slug, depth: 2, limit: 12 })` payload, and incoming impact checks via `ontology-atlas blast-radius <slug> [vault] --depth 2 --direction incoming` plus the MCP `blast_radius` payload. The brief embeds the full shared post-change ontology sync packet under the post-repair sync gate, including run conditions, the 14-check runtime graph DB gate, `health` / `cycles` / `growth_plan` / `maintenance_plan` / `validate_vault` MCP checks, CLI fallbacks, and skip cases. The visible Health target is labeled as the current repair target, then exposes the MCP node profile copy action directly beside the `/ontology/edit` and `/ontology` links, so Claude Code / Codex can inspect the target without expanding the secondary Actions panel. Impact and sync-gate copy actions remain in Actions because they are follow-up checks after the target is understood. The Health panel now shows the same repair order explicitly: inspect the target, repair ownership or source evidence in the builder, then run the shared sync gate. A visual graph-health read can therefore become either a portable review artifact for agents and collaborators or a focused MCP handoff payload.

Focus mode uses the same rule for its selected-node review brief. The copied brief keeps the topology / ontology / builder URLs and node profile / blast-radius checks readable for humans, then embeds the full sync packet instead of a one-line shorthand so the same artifact can be pasted directly into Claude Code / Codex after a non-trivial concept or frontmatter change.

It names the current analysis mode, shows a short next-action prompt, and keeps node / relation / issue metrics visible without expanding the heavier Sigma controls panel. Its Path prompt uses the same source-click then target-click workflow as the Sigma-side primer. When `pathFrom` and `pathTo` are already present in URL state, the bar resolves both graph IDs into readable titles and states the completed source-target route instead of asking for another target. The resolved Path state now exposes a compact proof-order panel, a compact path evidence copy action, and direct MCP copy actions for `path`, schema-aware `relation_check`, `explain_relation`, bounded `query_plan(all_paths)`, and bounded `all_paths` execution next to source / target handoff links into `/ontology` and `/ontology/edit`. The proof-order panel now includes a visible checklist: the shortest path is only a ready clue, while relation preflight, relation explanation, and bounded traversal are required before using the route as write evidence, and the post-write sync gate is marked as after-write work. The compact evidence includes the current topology URL, endpoint browse and builder URLs, the CLI path check, MCP `path`, inferred relation preflight reason, MCP `relation_check`, MCP `explain_relation`, bounded `query_plan(all_paths)`, bounded `all_paths`, the all_paths evidence contract, the same proof checklist, and the shared post-write sync gate, so a restored route can move immediately into Claude Code, Codex, collaborator review, or frontmatter-backed repair without relying only on the Sigma banner.

In Health mode, the bar now breaks the issue count into stale, orphan, and promotion chips, then exposes a compact inspect button, an `/ontology` browse link, a builder repair link, and a one-line next action for the first actionable bucket. That makes the graph-health overlay actionable at a glance instead of forcing users to infer every repair class from color alone.
