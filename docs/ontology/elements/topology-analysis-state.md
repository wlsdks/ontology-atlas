`src/views/home/model/url-state.ts` and `src/views/home/lib/topology-analysis.ts` persist `/topology?mode=<overview|focus|path|health>` and derive the small read model used by the analysis bar.

The state is URL-addressable so Claude Code, Codex, and humans can share the same topology analysis context. `HomePage` forwards Path mode into `SigmaTopology`, which makes the shortest-path workflow visible directly on the graph instead of leaving it as only text in the top bar. While Path mode is active, normal node clicks are interpreted as source / target picks; outside Path mode, Shift+click keeps the same workflow available without changing ordinary selection.

The Path workflow also derives relation labels for each hop from Sigma edge metadata. Render grouping (`kind`) stays separate from user-facing `relationType`, so the banner can explain whether the route is `depends_on`, `contains`, `describes`, or another vault relation without changing the visual edge reducer.

`src/widgets/topology-map-sigma/lib/path-interaction.ts` also formats the completed route as a markdown evidence brief. The copy payload includes hop count, relation-labeled route, graph node IDs, the current URL, source/target `/ontology?node=...` browse links, source/target `/ontology/edit?node=...` builder focus links, a CLI `oh-my-ontology path <source> <target> [vault] --max-hops 5` reproduction command, and the matching executable MCP `query_ontology({ operation: "path", from, to, maxHops: 5 })` call so the graph proof can move from `/topology` into agent handoff, collaborator review, or a concrete builder repair loop. It now also carries `explain_relation` follow-ups for CLI and MCP, so the shortest route can be paired with relation explanation before it is used as write, refactor, or coupling evidence. It also carries a bounded traversal-completeness follow-up: `oh-my-ontology all-paths <source> <target> [vault] --plan --max-hops 5 --limit 10 --search-budget 1000`, plus MCP `query_plan(all_paths)` and `all_paths` calls. That keeps a shortest visible route from being overclaimed as complete graph evidence before an agent reports `pathsComplete`, `totalPathsExact`, and budget status. The copied evidence also names the `all_paths` result contract fields to report before using the route as write evidence and embeds the full shared post-change sync packet under the post-write sync gate. That packet includes run conditions, MCP checks (`health`, `cycles`, `growth_plan`, `maintenance_plan`, `validate_vault`), CLI fallbacks, and skip cases, so a path-driven builder repair naturally closes with the same graph verification loop as relation confirmation. The result banner exposes dedicated MCP copy actions for the shortest `path`, relation preflight, `explain_relation`, bounded `all_paths` query planning, and bounded `all_paths` execution calls, matching the separate human-brief versus agent-call pattern used by Health mode and relation write preflight while making the plan-before-enumeration rule directly clickable.

Path analysis state now persists source and target node IDs in the home route query as `pathFrom` / `pathTo` while `mode=path` is active. `SigmaTopology` restores that route on mount, so a copied topology URL reopens the same visible path evidence instead of only restoring the analysis mode. The restore path accepts both Sigma graph IDs (`domain:views`) and vault slugs (`domains/views`, `capabilities/mcp-server`, including `ontology/`-prefixed evidence IDs), so URLs emitted by the builder relation confirmation and collaborator handoff packets land on the same visible Path workflow. The copied evidence brief also names the source and target explicitly before the relation-labeled route and includes a schema-aware relation preflight plus the inference reason aligned with the builder relation proposal, which makes the restored URL easier to paste into agent handoff, write review, or collaborator review. The result banner now shows that preflight relation type and reason before copy, and it pairs that with a visible `all_paths` completeness reminder so the graph workspace itself explains which `relation_check` and traversal verification should run.

The summary model also carries a Health breakdown for stale, orphan, and promotion counts plus a first actionable health target. Project signals are merged with `src/entities/knowledge-graph/lib/ontology-health-signals.ts`, which derives orphan and high fan-in review candidates from vault ontology nodes and edges while skipping epoch-0 vault sentinel dates for stale checks. The UI can therefore explain which graph-maintenance bucket the current issue total comes from and open the next target while keeping the Sigma overlay and URL mode in sync. When that target is an orphan, the copied health evidence now includes the same owner `relation_check` / `contains` preflight used by the `/ontology/insights` ownership repair packet, so topology health and insights handoffs ask Claude Code/Codex to prove the owner edge before writing frontmatter.

The state helper also formats a markdown Health evidence brief from the same summary model. That keeps the copied review artifact aligned with the counts and first inspect target shown in the analysis bar, including a deep link that carries the selected node through the existing `p` route query. When there is an actionable target, the formatter adds the bucket-specific next action plus a read-only CLI agent check so Claude Code, Codex, or a teammate can continue from the same node without reinterpreting the overlay.
