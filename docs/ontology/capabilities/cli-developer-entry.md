R14 에서 `import`, R15 follow-up 에서 graph-level 5 명령, R16-R17 에서 `analyze` / `infer-imports`, R+ cycle 에서 `path` / `orphans` + 두 `--apply` 플래그 + `bootstrap` 합본 명령, `compile`, `maintenance`, 그리고 설치된 MCP surface 를 확인하는 `mcp-verify` 명령 추가 — **총 27 명령**. 사용자가 vault 만든 후 *터미널 즉시* ontology 작업 가능 — 웹 UI / MCP 등록 불필요.

## 구현 단일 진실원

local commands 는 *cli 안* 구현 (4-way parser/3-way validator contract). graph-level + analyze/infer-imports + bootstrap + `--apply` 흐름은 *MCP server child_process spawn + JSON-RPC* — `cli/src/lib/mcp-call.mjs` 의 thin wrapper. wrapper 는 MCP `structuredContent` 를 먼저 사용하되, text JSON 과 `structuredContent` 가 둘 다 있으면 구조적으로 비교해 mismatch 를 실패로 처리하고 MCP tool name, 첫 mismatch path, parsed value, structuredContent value 를 에러에 포함한다. MCP spawn error / stdin write error / child process exit / child process signal / missing `tools/call` response 도 tool name / vault root / entry path / stderr 또는 stdout sample 을 포함해 agent 가 어떤 call 이 깨졌는지 바로 알 수 있게 한다. JSON-RPC 응답은 child `close` 이벤트에서 stdio drain 이후 파싱해 마지막 stdout/stderr chunk 를 놓치지 않는다. signal 종료는 missing-response fallback 이 아니라 `mcp terminated by SIGTERM` 같은 signal context 로 실패한다. JSON-RPC `error` 응답은 `message` 만 버블링하지 않고 `code` 와 `data` 도 보존하며, parse-time throw 가 Promise 를 미해결로 남겨 Node exit 13 으로 새는 경로를 막는다. one-shot call 은 기본 15s 후 fail-closed 로 reject 하고 `SIGTERM` 후 `SIGKILL` 을 예약해 `SIGTERM` 을 무시하는 MCP process 때문에 CLI 가 멈추지 않게 한다. 큰/느린 vault 는 `OMOT_CLI_MCP_TIMEOUT_MS=N`, timeout 후 cleanup window 는 `OMOT_CLI_MCP_KILL_GRACE_MS=N` 으로 조절한다. 성공 응답은 text 없이 `structuredContent` 만 있어도 수용하며, `structuredContent` 가 없는 경우에만 text JSON 으로 fallback 해 CLI 도 agent-facing MCP payload 계약을 그대로 dogfood한다. `analyze` / `bootstrap` 은 `analyze_repo_structure` top-level / 후보 배열 / evidence / skipped shape 을 먼저 검증하고, `infer-imports` / `bootstrap` 은 `infer_imports` top-level `rootPath` / import graph / `moduleEdges` shape 과 unresolved `reason` enum 을 먼저 검증한다. `analyze --apply` / `infer-imports --apply` / `bootstrap` 은 batch writer 응답 row count / row shape / top-level `postWriteMaintenance` shape 을 별도로 검증해 malformed `add_concepts` / `add_relations` 결과나 깨진 post-write maintenance payload 를 성공 요약으로 덮지 않는다. drift surface 0 (logic 복제 안 함). spawn ~50-100ms per call — bootstrap 은 3-4 회 호출이라 ~200-400ms 정도. `OMOT_MCP_PATH` explicit override 는 존재하지 않거나 파일이 아니면 fallback 없이 exit 2 로 실패해 잘못된 MCP entry 설정을 Node stack trace 나 조용한 monorepo fallback 으로 숨기지 않는다.

CLI batch summary 출력은 MCP row-level 실패가 `slug` / `from` / `to` / `type` 없이 반환되는 합법 케이스에서도 `undefined` 를 노출하지 않고 `concepts[n]` / `relations[n]` fallback label 로 표시한다. duplicate concept row 처럼 MCP error 자체가 `first seen at concepts[m]` 복구 힌트를 담은 경우에는 그 문자열을 그대로 보존해 agent 가 어떤 입력 행을 고칠지 잃지 않는다.

cli 가 별도 npm package — `oh-my-ontology` binary. cli/package.json 의 `dependencies: oh-my-ontology-mcp` 가 graph-level + apply + bootstrap 흐름 자동 활성. `cli/src/lib/cli-commands.mjs` 는 CLI command inventory / module runner registry / package description 의 command count 를 한 곳에서 노출하고 `cli/src/index.mjs` 의 runtime dispatch 도 같은 registry 로 실행해 command 추가 시 help / dispatcher / package metadata drift 를 줄인다. unknown top-level command 도 같은 registry 와 top-level aliases 에서 `complie` → `compile`, `hlep` → `help`, `--versoin` → `--version` 같은 closest-value hint 를 계산하고 실패 usage 는 stderr 로만 출력해 stdout 을 비워두므로 `pnpm dogfood:compile-fix` 는 docs/ontology 에 `compile --fix` 를 실행한 뒤 canonicalization 이 git diff 를 남기면 실패하고 성공 시 `[dogfood:compile-fix] docs/ontology unchanged` 요약으로 끝나며, `pnpm test:dogfood:args` 는 shared dogfood shortcut argument helper 를, `pnpm test:dogfood:compile-fix` 는 그 idempotence guard 를 spawn 없이 검증한다. `pnpm dogfood:health` 는 docs/ontology 의 `health --json` fail-closed health gate 만 빠르게 확인하고, `pnpm dogfood:brief` 는 docs/ontology 의 `workspace-brief --json` first-contact snapshot 만 빠르게 확인하며, `pnpm dogfood:status` 는 health 가 non-zero 여도 workspace-brief 를 계속 실행한 뒤 첫 실패 exit code 를 보존하고 `[dogfood:status] health:N · workspace-brief:N` 요약으로 끝나며, `pnpm test:dogfood:status` 는 그 shortcut 계약만 spawn 없이 검증하며, `pnpm dogfood:compile-fix -- --help` / `pnpm dogfood:status -- --help` 는 gate 를 실행하지 않는 usage path 이고 잘못된 인자는 exit 2 로 실패하며 `--help` 근접 오타는 `Did you mean --help?` 힌트를 준다. `pnpm dogfood:verify` 는 반복 dogfood vault 검증용 full gate 이다. `pnpm cli:mcp-verify docs/ontology --timeout-ms hint 는 `--vault <path>` 를 보존해 agent 가 다른 vault 로 재시도하지 않게 한다. MCP package 의 직접 `npm run verify -- --vault <path>` 경로도 같은 vault-preserving retry 계약을 공유한다.

`pnpm dogfood:status` 실패 출력은 `[dogfood:status] health:N · workspace-brief:N` child status 요약 뒤에 `pnpm dogfood:verify` follow-up hint 를 붙여 다음 escalation gate 를 바로 보여준다.

`pnpm test:dogfood:script-refs` 는 `scripts/lib/pnpm-script-refs.mjs` unit test, focused node-test wrapper 계약, `scripts/check-package-contracts.test.mjs` 의 filtered integration script discoverability 계약을 함께 실행해, `mcp-verify --help`, `dogfood:help`, root / MCP / CLI README, development checks 의 `pnpm ...` references 와 package.json script body 안의 chained `pnpm ...` references 가 실제 root package script 와 drift 나지 않는지 같은 방식으로 검증한다. `pnpm install` / `pnpm exec` 같은 pnpm builtin 과 `pnpm integration:*` 같은 wildcard 설명, `pnpm forwards` 같은 prose, `pnpm ...` placeholder 는 script reference 로 오인하지 않는다. `pnpm --silent dogfood:status` / `pnpm --filter ./mcp test:mcp:package` 처럼 pnpm option 이 script 앞에 붙은 예시도 option token 이 아니라 실제 script 를 검증한다. `pnpm run dogfood:status` / `pnpm --silent run --if-present test:mcp:docs` 처럼 `run` 을 명시한 예시도 `run` 이 아니라 실제 package script 를 검증한다. `(pnpm test:mcp:docs)` / `pnpm "dogfood:status"` 처럼 subshell 이나 quoted script 로 적힌 예시도 실제 script 이름으로 정규화한다. `scripts/run-focused-node-test.mjs` 는 Node `--test-name-pattern` 이 0개 테스트와 매칭되어도 exit 0 이 되는 경로를 `# pass 0` summary 로 감지해 fail-closed 처리하고, pattern 없는 wrapper 호출이나 target 없는 wrapper 호출도 exit 2 로 거부해 accidental full-suite/default-discovery run 을 막으며, `--test-concurrency 1` 또는 `--test-timeout 1000` 같은 Node test option value 를 target 으로 오인하지 않고, split option 값이 빠져도 다음 option value 를 target 으로 새지 않게 한다. 성공 시에도 `[focused-node-test] pattern=... targets=... tests=N pass=N fail=0 cancelled=0 skipped=N` 요약을 끝에 남겨 focused gate 가 실제 몇 개 테스트를 실행했고 얼마를 skip 했는지 긴 TAP 출력 끝에서 바로 확인하게 한다. 실패 메시지에 pattern 과 test target 파일을 함께 보여주므로 root 의 focused integration subset / `test:mcp:*` package script 가 조용히 전부 skip 되지 않는다. wrapper 는 default TAP summary 의 `# tests` / `# pass` / `# fail` / `# cancelled` counts 를 요구하므로 argv 또는 `NODE_OPTIONS` 의 custom reporter 옵션은 spawn 전에 거부한다. package contract 는 `--test-name-pattern` 을 쓰는 모든 root script 를 스캔해 반드시 이 wrapper 를 거치도록 고정하고, wrapper 를 쓰는 script 가 명시적 pattern 과 target 을 넘기는지도 고정한다.
