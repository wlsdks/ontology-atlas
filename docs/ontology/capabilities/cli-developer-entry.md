# CLI Developer Entry

R12 (2026-05-04) 에 도입된 *developer-primary* 진입점. R14 에서 `import`, R15 follow-up 에서 graph-level 5 명령, R16-R17 에서 `analyze` / `infer-imports`, R+ cycle 에서 `path` / `explain` / `all-paths` / `reachability` / `orphans` + 두 `--apply` 플래그 + `bootstrap` 합본 명령, `compile`, `growth`, `maintenance`, graph DB-style scan 인 `match-nodes` / `match-edges`, domain coupling matrix 를 보는 `domain-matrix`, write 전 schema preflight 인 `relation-check`, Claude Code/Codex handoff 를 보는 `agent-brief`, 그리고 설치된 MCP surface 를 확인하는 `mcp-verify` 명령 추가 — **총 36 명령**. 사용자가 vault 만든 후 *터미널 즉시* ontology 작업 가능 — 웹 UI / MCP 등록 불필요.

`pnpm smoke:memory-loop` 는 fresh repo 에서 `init -> bootstrap -> validate -> workspace_brief -> agent_brief -> node_profile -> sync proposal` 까지 10분 예산 안에 이어지는지 검증한다. 새 feature 파일을 만든 뒤 `analyze_repo_structure` 가 side-effect-free 후보를 제안하고 git diff 와 일치하는지 확인해, agent 가 vault 쓰기 전에도 사용자가 sync 제안을 리뷰할 수 있게 한다.

## 구현 단일 진실원

local commands 는 *cli 안* 구현 (4-way parser/3-way validator contract). graph-level + analyze/infer-imports + bootstrap + `--apply` 흐름은 *MCP server child_process spawn + JSON-RPC* — `cli/src/lib/mcp-call.mjs` 의 thin wrapper. wrapper 는 MCP `structuredContent` 를 먼저 사용하되, text JSON 과 `structuredContent` 가 둘 다 있으면 구조적으로 비교해 mismatch 를 실패로 처리하고 MCP tool name, 첫 mismatch path, parsed value, structuredContent value 를 에러에 포함한다. MCP spawn error / stdin write error / child process exit / child process signal / missing `tools/call` response 도 tool name / vault root / entry path / stderr 또는 stdout sample 을 포함해 agent 가 어떤 call 이 깨졌는지 바로 알 수 있게 한다. JSON-RPC 응답은 child `close` 이벤트에서 stdio drain 이후 파싱해 마지막 stdout/stderr chunk 를 놓치지 않는다. signal 종료는 missing-response fallback 이 아니라 `mcp terminated by SIGTERM` 같은 signal context 로 실패한다. JSON-RPC `error` 응답은 `message` 만 버블링하지 않고 `code` 와 `data` 도 보존하며, parse-time throw 가 Promise 를 미해결로 남겨 Node exit 13 으로 새는 경로를 막는다. one-shot call 은 기본 15s 후 fail-closed 로 reject 하고 `SIGTERM` 후 `SIGKILL` 을 예약해 `SIGTERM` 을 무시하는 MCP process 때문에 CLI 가 멈추지 않게 한다. 큰/느린 vault 는 `OMOT_CLI_MCP_TIMEOUT_MS=N`, timeout 후 cleanup window 는 `OMOT_CLI_MCP_KILL_GRACE_MS=N` 으로 조절한다. 성공 응답은 text 없이 `structuredContent` 만 있어도 수용하며, `structuredContent` 가 없는 경우에만 text JSON 으로 fallback 해 CLI 도 agent-facing MCP payload 계약을 그대로 dogfood한다. `analyze` / `bootstrap` 은 `analyze_repo_structure` top-level / 후보 배열 / evidence / skipped shape 을 먼저 검증하고, `infer-imports` / `bootstrap` 은 `infer_imports` top-level `rootPath` / import graph / `moduleEdges` shape 과 unresolved `reason` enum 을 먼저 검증한다. `analyze --apply` / `infer-imports --apply` / `bootstrap` 은 batch writer 응답 row count / row shape / top-level `postWriteMaintenance` shape 을 별도로 검증해 malformed `add_concepts` / `add_relations` 결과나 깨진 post-write maintenance payload 를 성공 요약으로 덮지 않는다. drift surface 0 (logic 복제 안 함). spawn ~50-100ms per call — bootstrap 은 3-4 회 호출이라 ~200-400ms 정도. `OMOT_MCP_PATH` explicit override 는 존재하지 않거나 파일이 아니면 fallback 없이 exit 2 로 실패해 잘못된 MCP entry 설정을 Node stack trace 나 조용한 monorepo fallback 으로 숨기지 않는다.

CLI batch summary 출력은 MCP row-level 실패가 `slug` / `from` / `to` / `type` 없이 반환되는 합법 케이스에서도 `undefined` 를 노출하지 않고 `concepts[n]` / `relations[n]` fallback label 로 표시한다. duplicate concept row 처럼 MCP error 자체가 `first seen at concepts[m]` 복구 힌트를 담은 경우에는 그 문자열을 그대로 보존해 agent 가 어떤 입력 행을 고칠지 잃지 않는다.

cli 가 별도 npm package — `oh-my-ontology` binary. cli/package.json 의 `dependencies: oh-my-ontology-mcp` 가 graph-level + apply + bootstrap 흐름 자동 활성. `cli/src/lib/cli-commands.mjs` 는 CLI command inventory
