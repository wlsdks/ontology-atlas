title: Topology Analysis Modes
domain: views

---

`/topology` exposes explicit Overview / Focus / Path / Health modes so the Sigma graph reads as an analysis workspace instead of a decorative map. The mode switcher remains reachable on mobile, so small-screen topology is not reduced back to a passive map. Path prompts and results are offset below the mobile mode switcher so the click-through workflow remains legible on narrow screens.

Each mode changes the active overlays and controls instead of only changing copy: Overview exposes a copyable first-contact graph handoff, Focus narrows depth and highlights backrefs, Health turns on audit highlights, and Path keeps the graph ready for shortest-path proof. The Overview brief includes graph counts, health buckets, Health and Insights handoff URLs, `oh-my-ontology overview [vault] --limit 5`, executable MCP `query_ontology({"operation":"overview"})`, MCP `query_plan(targetOperation: "overview")`, and workspace-brief checks so Claude Code / Codex can start with the same map the collaborator sees. When Focus mode has a selected node, the analysis bar now shows the focused review order before any edit handoff: read the node profile, trace incoming impact, edit or confirm the concept meaning, then run the sync gate after a change. The same bar can copy a focused review brief with topology, ontology, builder, CLI, MCP, impact, and post-change sync-gate handoffs, copy only that node's MCP `node_profile`, incoming `blast_radius`, or sync-gate calls, or open the selected concept directly in `/ontology` or `/ontology/edit`. A visual neighborhood can therefore move directly into Claude Code / Codex inspection, collaborator vocabulary review, or builder repair without forcing the user through Health mode, and a builder edit that follows the focus review can close with the same graph verification loop. When Path mode is active, the analysis bar and Sigma primer both tell the user to click a source node and then click a target; Shift+click remains a fallback outside Path mode. The existing path result banner then shows the hop chain and lets the user jump to any node in that chain.

Path results now preserve the relation key between each hop in the banner, so a route reads as `node --relation--> node` instead of just a sequence of names. Project dependency edges expose `depends_on` / `contains`, and vault ontology edges expose their frontmatter relation type, making the visible route useful as evidence for developer review or agent handoff.

The result banner can copy a markdown path evidence brief with hop count, explicit source and target rows, a relation-labeled route, graph node IDs, and the current topology URL. The URL stores `pathFrom` / `pathTo` while Path mode is active, so opening the link restores the same source-target route instead of only reopening the generic Path workspace. Restored endpoints may be either graph IDs or vault slugs, which means a `/ontology/edit` relation write packet can hand off `/topology/?mode=path&pathFrom=domains%2Fviews&pathTo=capabilities%2Ftopology-analysis-modes` and still restore the same visible route. The analysis bar resolves those two IDs back into readable source and target titles, making a restored route visibly complete before the user inspects or copies the graph banner. It now also exposes its own compact path evidence copy action plus direct MCP copy actions for shortest `path`, schema-aware `relation_check`, `explain_relation`, and bounded `query_plan(all_paths)`, so the restored route remains agent-ready even when the Sigma result banner is off-screen or visually crowded. The bar shows a small proof-order panel that tells the user to treat the visible path as a clue, then run relation preflight, relation explanation, and bounded traversal planning before using the route as write evidence. That panel now renders the proof order as a compact checklist: visible path clue is ready, relation preflight / relation explanation / bounded traversal are required, and the post-write sync gate is after-write work. The compact bar packet keeps endpoint URLs, builder handoffs, the CLI path check, MCP `path`, the inferred relation preflight reason and executable `relation_check`, executable `explain_relation`, bounded `query_plan(all_paths)`, bounded `all_paths`, the all_paths evidence contract, the same proof checklist, and the shared post-write sync gate. The richer Sigma evidence packet includes the same relation preflight using the same schema-aware key inference as the builder (`domain → capability` checks `capabilities`, `capability → element` checks `elements`, ambiguous pairs fall back to `relates`). The banner shows the inferred preflight type and reason inline, then pairs it with an `explain_relation` follow-up and a visible `all_paths` completeness reminder. The copied packet includes the CLI and executable MCP `query_ontology(...)` `explain_relation` checks, the bounded CLI `all-paths --plan` command, MCP `query_plan(all_paths)`, MCP `all_paths`, and the result contract fields an agent must report before using paths as write evidence. It also embeds the full shared post-change sync packet under the post-write sync gate, so a path-driven builder repair naturally closes with run conditions, MCP health checks, CLI fallbacks, skip cases, and the same graph verification loop as relation confirmation. The banner also exposes dedicated MCP copy actions for shortest `path`, relation preflight, `explain_relation`, bounded `all_paths` query planning, and bounded `all_paths` execution, and each copies the same `query_ontology(...)` call form used by `/ontology` and `/ontology/insights`. Claude Code / Codex can therefore explain the relation, plan traversal before enumerating paths, and report path completeness before treating the visible shortest route as proof. This turns an on-canvas route into something a developer can paste into Claude Code, Codex, a PR note, or a collaborator review without re-explaining the graph manually.

The Path workflow is covered by browser evidence for graph-ID restored routes, builder-style vault-slug restored routes, and the mobile primer layout below the analysis bar, so the mode remains a real interaction workflow rather than only a query-string parser or visual label.

Health mode now pairs the audit overlay with an explicit stale / orphan / promotion breakdown in the analysis bar and a first actionable inspect target. The target prioritizes stale nodes, then orphans, then promotion candidates, and it includes both project rows and vault-derived ontology nodes so dogfood topology health is not limited to the single project shell. Browser coverage opens `/topology?mode=health` and checks the active Health mode, the visible health prompt, the stale/orphan/promotion chips, and the copyable health evidence action.

Health results can also be copied as a markdown evidence brief with issue totals, bucket counts, the first inspect target, the current topology URL, an inspect URL that opens Health mode with the target node selected, an `/ontology` browse URL, a builder repair URL, node-profile checks, orphan-owner `relation-check <owner-slug> <target> contains` preflight when the target is unowned, incoming `blast_radius` impact checks, and the post-repair sync gate sequence. The visible Health target now exposes both the browse link and builder repair link before the copy actions, so a health review can read the concept before changing frontmatter. The bar also exposes dedicated MCP impact and sync-gate copy actions for the same target. This makes graph-health review portable into Claude Code, Codex, PR notes, or collaborator vocabulary review instead of leaving it as a visual-only overlay, and it tells the next agent how to close the graph after a builder repair.

This ties the analysis bar, URL state, and actual graph interaction together so path work and graph-health review are visible enough for developer review, agent handoff, and collaborator explanation.
