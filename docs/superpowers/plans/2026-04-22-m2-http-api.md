# M2 · 외부 HTTP API (`POST /api/v1/docs`)

**시작일:** 2026-04-22
**선행:** P0-B Phase 0–6 완료. write 경로가 컨테이너 인지 (`upsertProjectInContainer`, `createProjectAdaptive`).
**목표:** 외부 도구 (CLI · CI · MCP · 기타 클라이언트) 가 HTTP 한 방으로 워크스페이스에 문서/노드를 push 하고, 받은 직후 토폴로지에 반영되게.

## 야심 정렬

이 endpoint 가 들어와야:
- 개발자가 IDE 에서 `npx project-narnia push docs/iam-spec.md` → 그 즉시 narnia 컨테이너의 IAM 노드 생성
- CI 파이프라인이 PR merge 시 `curl -H "Authorization: Bearer ..." -d @doc.json` → 자동 등록
- M3 presence 가 "이 노드 작업 중" payload 도 동일 패턴으로 push
- MCP 서버 (`project-narnia-mcp`) 가 같은 endpoint 를 wrapper 로 호출

## 비-목표 (이번 마일스톤 외)

- 지식 추출 (knowledge subsystem) 은 별도 파이프라인 — POST 가 받은 raw md 를 자동 추출 큐에 올리는 hook 은 Phase 4+ 로 분리
- 다중 문서 batch endpoint — 우선 단건만, batch 는 후속
- WebSocket / SSE — pull 은 기존 onSnapshot 구독으로 해결, push (presence) 는 M3 별도 채널

## 엔드포인트 contract (v1)

```
POST /api/v1/docs
Content-Type: application/json
Authorization: Bearer <api_key>

{
  "accountId": "stark",            // required — 어느 워크스페이스
  "projectId": "narnia",           // optional — 미지정 시 "general"
  "doc": {
    "slug": "iam-hub",             // required — kebab-case, 컨테이너 내 unique
    "name": "IAM Hub",             // required
    "isHub": true,                 // optional — 기본 false
    "category": "platform",        // optional — taxonomy 미정 시 fallback
    "status": "developing",        // optional — taxonomy 미정 시 fallback
    "description": "...",          // optional
    "detail": "...",               // optional, markdown
    "tags": ["auth", "edge"],      // optional
    "stack": ["typescript"],       // optional
    "dependencies": ["reactor"],   // optional — 의존 다른 노드 slug
    "hubIds": ["iam-hub"],         // optional — node 일 때 소속 hub
    "links": [
      { "label": "Repo", "url": "..." }
    ],                              // optional
    "owner": "stark",              // optional
    "icon": "🛡",                   // optional
    "screenshots": [],             // optional — 외부 URL 또는 storage path
    "timeline": {                  // optional
      "startedAt": "2026-04-01",
      "launchedAt": "2026-05-01"
    },
    "progress": 30,                // optional 0~100
    "metadata": {                  // optional — 자유 키 (presence 등 확장용)
      "currentEditor": "stark",
      "lastPushAt": "2026-04-22T09:30:00Z"
    }
  }
}
```

**Response (200):**
```json
{
  "status": "ok",
  "action": "created",   // "created" | "updated"
  "path": "accounts/stark/workspaceProjects/narnia/hubs/iam-hub",
  "containerCreated": false,  // ?pj 미존재 시 자동 생성했으면 true
  "writtenAt": "2026-04-22T09:30:01.234Z"
}
```

**Errors:**
- `401 unauthorized` — missing/invalid API key
- `403 forbidden` — 키는 유효하나 해당 accountId 권한 없음
- `400 invalid_argument` — required 필드 누락 또는 형식 오류
- `409 conflict` — slug 충돌 (현재는 upsert 라 발생 안 하지만 future 예약)
- `429 rate_limited` — 분당 호출 한도 초과
- `500 internal` — 서버 오류

## 인증 모델

**API Key 발급:** admin UI (`/admin/api-keys/`) 에서 발급. accountId 별 N개 키 가능. 키는 한 번만 표시되고 hashed 저장.

**키 검증:**
- `Authorization: Bearer <key>` 헤더에서 키 추출
- DB 에서 hash 비교
- 키 → accountId 매핑 → request.body.accountId 와 일치 확인 (다르면 403)
- `lastUsedAt`, `usageCount` 갱신 (rate limit 용)

**키 권한 범위:**
- v1: 단일 accountId 에 묶임. 그 안의 모든 컨테이너에 read/write 가능.
- 미래: per-container, per-action (read-only / write-only) scope 분할.

## Firestore 스키마 (신규)

```
accounts/{accountId}/apiKeys/{keyId}
  - id: string
  - name: string                    # 사용자 식별용 ("MCP from laptop", "CI bot")
  - keyHash: string                 # SHA-256(key)
  - keyPrefix: string               # 키 처음 8자 (UI 표시용 — 전체키 표시 안 함)
  - scope: "account-rw"             # v1 단일 scope
  - createdAt: Timestamp
  - createdBy: string (email)
  - lastUsedAt: Timestamp | null
  - usageCount: number
  - revokedAt: Timestamp | null
```

**Rules:**
- `read/list`: account member 또는 admin (UI 에서 키 목록 보기)
- `create`: account member 또는 admin
- `update`: 자기 자신의 revoke 만 (revokedAt set)
- `delete`: false — revoke 후 audit 보존

## 단계별 플랜

### Phase 0 — 이 설계 문서 ✅

### Phase 1 — API Key entity · 발급 UI
- `src/entities/api-key/` — types, mapper, api (CRUD)
- `firestore.rules` 의 `apiKeys` 추가
- `DATA-MODEL.md` 갱신
- `/admin/api-keys/` 페이지 — 발급 / 목록 / revoke
- 키 생성: `crypto.randomUUID() + nanoid` 형태 32자 무작위. 발급 직후 한 번만 평문 노출 ("키를 안전한 곳에 복사하세요").

### Phase 2 — HTTP function `receiveDoc`
- `functions/api-receive-doc.js` (또는 `index.js` 에 추가)
- `onRequest` (REST) — 기존 onCall 과 다른 endpoint
- 키 헤더 검증 → accountId match 확인 → body validation → upsert
- write 경로:
  - projectId 명시 OR runtime URL 추적 불가하므로 명시 → `accounts/{accountId}/workspaceProjects/{projectId|"general"}/hubs|nodes/{slug}`
  - isHub 로 hubs/nodes 분기 (frontend `upsertProjectInContainer` 와 동일 로직 사용)
  - target 컨테이너 doc 없으면 자동 생성 (containerCreated=true 반환)
- rate limit: 분당 60 (간단 in-memory 또는 Firestore 기반)
- CORS: `Access-Control-Allow-Origin: *` (API key 만 있으면 어디서든 호출 가능해야 함)

### Phase 3 — npm 패키지 `project-narnia-client`
- `packages/narnia-client/` — TypeScript 클래스 SDK
- `new Narnia({ baseUrl, apiKey, accountId, projectId? })`
- `await client.pushDoc({ slug, name, ... })` → POST 호출 wrapper
- error mapping (status code → typed error)
- README + 사용 예시

### Phase 4 — MCP 패키지 `project-narnia-mcp`
- `packages/narnia-mcp/` — MCP server 구현
- Claude Code · Cursor 가 tool 로 인식할 수 있는 표준 MCP API
- `narnia.push_doc(...)` tool → 내부적으로 narnia-client 사용
- Tool definitions: push_doc, list_projects, get_node, update_metadata
- `npx -y project-narnia-mcp` 로 즉시 실행 가능하게 bin 설정

### Phase 5 — extraction hook (선택)
- POST 받은 직후 `doc.detail` (markdown) 이 있으면 자동으로 knowledge extraction 큐 enqueue
- 같은 `enqueueExtractionJob` 흐름 재사용
- 키 권한 체크 + 화이트리스트 (extractor:enabled scope)

### Phase 6 — observability + admin
- `/admin/api-keys/` 의 각 키 마지막 사용/카운트 표시
- 실패 로그 별도 컬렉션 (`apiKeyAuditLog`) — 무효 키 시도 / 권한 거부 / rate limit 초과
- 일/월 사용량 차트

## 위험·완화

- **API key 유출**: hashed 저장 + 한 번만 평문 노출 + 즉시 revoke 가능
- **남용**: rate limit + per-key 일일 한도 (Phase 6)
- **비용 폭증**: 각 호출은 Firestore write 1건 + read 0~2건. 분당 60 한도면 일 86,400 write/key. 알람 추가 필요
- **schema drift**: v1 endpoint 는 v1 schema 만 수신. v2 추가 시 별도 path
- **CORS 남용**: 키 검증 후만 동작이므로 `*` 허용해도 안전. 다만 OPTIONS preflight 처리 주의
